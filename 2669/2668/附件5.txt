附件5

数字化审计模型：数字化营销审计智能分析与穿透预警模型

一、模型目标

针对传统营销审计中“数据来源分散、制度条文繁多、人工抽查效率偏低、审前风险线索不清晰”等痛点，构建一套覆盖营销全流程、全量数据的数字化审计分析模型，提升营销审计的自动化、标准化和穿透式分析能力。

本模型的建设目标主要包括三个方面：

一是实现全量数据的自动接入与快速审计。通过整合 DMP 营销系统、SAP 财务系统、审计附表及补充台账数据，自动完成数据清洗、字段映射、主数据匹配、规则扫描和结果输出，减少人工筛查工作量。

二是推动制度规则向可执行逻辑转化。围绕营销管理、区域授权、风险分级、客户管理、投标管理等制度要求，将核心审计判断口径沉淀为结构化规则库，形成“制度条文 - 审计规则 - 输出结论”的闭环。

三是提升跨模型穿透分析能力。以“制度红线识别 + 异常预警提示 + 关联线索穿透”为核心思路，支持在单模型识别基础上形成跨模型风险关联提示，为审计人员提供问题排查方向和重点核查对象。

需要说明的是，本模型中的跨模型关联分析主要用于提示风险相关性和核查方向，不直接替代审计结论中的因果认定。

二、模型功能简介

本模型按照“数据导入 -> 数据治理 -> 模型扫描 -> 关联分析 -> 结果输出”的流程运行，支持通过网页端集中完成批量审计分析。结合当前代码实现，系统主要功能包括：

1. 多源数据接入与清洗。当前网页端已支持 4 类数据槽位上传，其中 DMP 销售数据、审计附表和中标报量为必传文件，SAP 财务数据为可选文件。系统在执行前完成基础文件接收、格式校验和落盘。

2. 模型化规则扫描。当前代码已注册 3 个维度、11 个模型，支持用户在网页端勾选模型后，调用主程序批量运行。

3. 风险关联提示。对部分重点场景形成跨模型联动分析思路，辅助审计人员从单点问题进一步追踪相关风险链条。需要说明的是，当前代码层尚未形成独立的“关联链结果表”和“关联链页面视图”，该部分仍应定位为分析框架和后续增强方向。

4. 结果标准化输出。当前系统已具备模型统计结果、单模型明细、综合 Markdown 报告和网页可视化展示能力。

结合现有实现情况，系统已具备“上传文件 -> 选择模型 -> 调用 main.py 执行 -> 缓存结果 -> 页面展示”的基础闭环。考虑到当前数据基础和主数据映射仍在完善中，现阶段建议优先聚焦业务价值高、规则边界清晰、数据可获得性较强的重点模型，逐步深化，而非一次性追求所有模型同时做深做全。

三、技术及数据说明

（一）技术说明

本模型基于 Python 开发，核心使用 Flask、pandas、openpyxl 等组件，整体采用“数据适配 -> 模型检测 -> 关联提示 -> 报告输出 -> Web 展示”的分层架构。

结合现有代码实现，系统后端主要由 `web_app.py` 提供网页服务和接口能力，通过调用 `main.py` 执行选定模型；运行结果以缓存文件和报告文件形式输出到 `output` 目录。前端当前已实现以下页面能力：

1. 首页：上传文件、选择模型、启动任务、查看执行状态。
2. 问题页：展示聚合后的问题统计信息。
3. 模型页：查看单模型明细结果。
4. 报告页：展示由 Markdown 转换后的综合报告。

因此，方案中关于“轻量化 Web 作业台”的描述是与代码一致的；但若表述为“完整风险中台、关联链图谱或客户健康驾驶舱”，则超出现阶段实现能力，建议保留为后续规划而非当前成果表述。

需要补充说明的是，规则库并非简单摘取制度原文，而应经过以下流程固化：

1. 制度原文梳理。
2. 审计口径提炼。
3. 业务部门确认。
4. 必要时经法务或制度管理部门复核。
5. 转化为系统规则并形成版本管理记录。

其中，对“11 条严禁投标红线规则”等关键规则，建议在正式实施时形成《规则定义清单》，明确规则来源条款、适用范围、判断逻辑、责任确认人和启用版本。

（二）数据说明

模型主要使用以下数据来源：

1. DMP 营销系统数据。
主要包括项目地址、申报单位、签约额、利润率、付款条件、客户信息、投标方式、合同条款标签等。

2. SAP 财务系统数据。
主要包括实际产值、实际利润、累计收款、累计付款、资金余额、保证金、预收款、应收回款等。

3. 审计附表及补充台账。
主要用于补充 DMP 缺失字段，如项目真实状态、停工退场原因、客户分级、主责单位、特殊审批情况等。

4. 区域授权清单、制度规则清单、客户名录等基础资料。
用于支撑区域识别、制度判断和客户结构分析。

需要重点说明以下三个数据治理前提：

第一，67 家单位授权清单必须建立动态维护机制。应明确维护责任部门、更新频率、版本生效日期及历史版本追溯机制。否则，区域窜区识别结果可能出现误判。

第二，DMP、SAP、营销侧编码之间的主数据映射必须单独建设和持续治理。对于“60 位编码、12 位编码、C 开头编码”等不同编码体系，应建立统一项目主键和映射规则表，避免跨系统比对失真。

第三，当前代码层面对 SAP 财务数据槽位的设置仍为“可选”，这意味着部分依赖财务数据的模型在数据不完整时可能只能输出有限结论。因此，正式运行时应为各模型建立“前置数据校验规则”，明确哪些模型在 SAP 缺失时禁止执行，哪些模型允许降级运行。

四、模型逻辑及步骤

（一）总体逻辑

模型总体遵循“数据接入 -> ETL 清洗 -> 主数据对齐 -> 模型扫描 -> 关联提示 -> 审计输出”的处理流程。

系统首先完成多源数据接入与标准化处理，在此基础上通过统一项目主键实现跨系统关联，再按照规则模型逐项扫描，最后生成问题结果和关联提示信息。

与传统人工审计相比，本模型的核心价值不在于替代审计判断，而在于提高全量筛查效率、统一口径、提供重点线索，并辅助审计人员开展进一步核查。

（二）数据治理前置层

在模型运行前，先执行数据治理前置层，主要包括：

1. 字段标准化与空值校验。
2. 项目编码映射与主数据关联。
3. 跨系统时间、金额、状态一致性检查。
4. 对已签约未开工、明显无效数据、重复记录等进行基础过滤。
5. 输出数据异常标记，供后续模型结果解释时回溯使用。

该前置层的作用，是为后续模型提供相对干净、可比、可追溯的数据基础。

（三）模型分维度设计

本模型分为三个维度，共 11 个模型。该模型数量与当前网页端注册模型清单一致：

维度一：营销战略与布局模型组

模型 1.1：区域布局偏差与窜区检测
通过比对项目地址、签约单位与区域授权清单，识别跨区域承接、非常规区域承接等风险项目。

模型 1.2：业务结构偏差检测
分析各单位业务板块结构、重点业务占比、行业集中度等指标，识别偏离战略导向的结构性风险。

模型 1.3：战略客户布局分析
分析战略客户合作连续性、客户贡献波动、客户集中度等情况，识别重点客户维护不足或依赖过高问题。

模型 1.4：营销统计数据多维交叉验真
通过时间线、金额逻辑、台账一致性和异常波动识别数据质量风险，为后续模型提供可信度参考。

维度二：合同质量与风险穿透模型组

模型 2.1：风险分级红线检测
围绕严禁投标红线规则，对客户背景、付款条件、垫资要求、担保安排等进行规则扫描，识别高风险承接项目。

模型 2.2：盈利偏差与履约异常检测
比对投标测算利润率与实际利润率，识别盈利预测偏差、长期未实质开工、履约推进异常等问题。

模型 2.3：资金安全监测
识别保证金、预收款、应收回款等方面的逾期和异常风险，关注客户退出后形成的资金敞口。

模型 2.4：合同条款风险检测
识别不平等条款、无限责任、停工权风险、证照手续不完备等合同层面风险。

模型 2.5：施工真实性验证
针对开工后短期停工退场、在施状态与产值回款不匹配等情况，识别“签约履约偏差”和疑似虚假施工问题。

维度三：客户健康度与绩效模型组

模型 3.1：客户全生命周期监测
分析中标转化、客户流失、长期无合作客户、客户评级失真等情况。

模型 3.2：新客户质量评估
分析优质新客户占比、合作持续性、结构优化空间等，辅助客户结构调整。

（四）三条核心关联链的修正表述

原模型设计提出了三条核心关联链。经优化后，建议将其统一定义为“跨模型风险关联提示链”，主要用于提示核查方向，不直接作因果归因。

结合当前代码实现，需要进一步说明：现阶段系统已经具备“单模型结果输出”和“综合报告输出”能力，但尚未形成专门的关联链落地机制，包括：

1. 尚无独立的关联链结果数据结构。
2. 尚无“链路命中对象 - 证据节点 - 风险标签”的统一输出表。
3. 尚无专门展示三条关联链的网页模块或图谱页面。

因此，这三条关联链在本方案中应准确表述为“跨模型分析框架”，其工程化落地仍需在结果层和前端展示层继续补强。

关联链 1：战略 -> 风险关联提示链

由模型 1.1 输出非常规区域项目和疑似窜区项目，再联动模型 2.1、2.2、2.5，对其是否同时存在高风险承接、盈利异常、施工真实性异常等情况进行关联提示。

该链条的输出建议表述为：
“非常规区域项目与高风险承接、盈利异常、施工异常存在叠加风险特征，建议作为重点核查对象。”

不宜直接表述为：
“区域违规必然导致风控失效和盈利恶化。”

关联链 2：客户 -> 合同质量关联提示链

由模型 1.3、3.1 输出客户流失预警、战略客户产出波动等信号，再联动模型 2.2、2.3，对关联项目的利润率变化、资金回收情况进行比对分析。

该链条的输出建议表述为：
“客户质量变化与项目盈利和资金回收风险存在联动迹象，建议关注客户价值管理的连续性。”

关联链 3：数据质量 -> 决策可靠性提示链

由前置过滤层和模型 1.4 输出数据异常标记，对其他模型结果进行可信度提示。当任一模型输出异常结果时，优先回溯数据质量标记，判断该异常更可能来自业务问题还是数据污染问题。

该链条的输出建议表述为：
“前置过滤层与模型 1.4 共同构成数据质量防护层，用于提升模型结果解释的可靠性。”

（五）规则来源和维护机制补充

对于区域授权、投标红线、客户分类、合同条款风险等关键规则，建议建立以下机制：

1. 规则来源登记。
每条规则须明确对应制度名称、条款编号、解释口径和适用范围。

2. 业务确认机制。
由审计、市场、法务或制度管理相关部门共同确认关键规则。

3. 版本更新机制。
制度修订后同步更新规则库，并保留历史版本。

4. 运行回溯机制。
模型输出结果可追溯至当期使用的规则版本和数据版本。

五、审计问题成果

结合现有代码，模型当前主要输出以下几类成果：

（一）制度合规类成果

1. 区域布局违规识别结果。
2. 高风险承接和红线触碰项目清单。
3. 合同条款不平衡及手续瑕疵问题清单。

（二）经营质量类成果

1. 盈利预测偏差项目清单。
2. 长期未实质开工或履约推进异常项目清单。
3. 保证金、预收款、回款逾期风险清单。

（三）客户健康类成果

1. 战略客户贡献波动分析。
2. 客户流失和长期无合作客户预警。
3. 新客户质量与客户结构优化建议。

（四）数据质量类成果

1. 主数据映射异常清单。
2. 时间线倒挂、金额失真、疑似规避评审等异常标记。
3. 对模型结果可信度的辅助说明。

（五）关联提示类成果

三条核心关联链的输出建议以“关联提示”形式呈现，包括：

1. 关联链命中对象。
2. 涉及模型及风险标签。
3. 关联证据摘要。
4. 建议进一步核查方向。

该类结果应作为审计线索辅助材料，不直接代替定性结论。

从当前代码实现看，以上成果中已经明确落地的主要是：

1. 模型统计信息。
2. 单模型问题明细。
3. 分类汇总结果。
4. 综合 Markdown 报告。

而“关联提示类成果”目前更多体现在方案层和分析逻辑层，尚未形成独立的系统化输出界面。

六、模型总结

本模型体现了审计思维与数字化技术结合的探索方向，具备较好的体系设计基础和业务延展空间。结合当前代码实现，可以判断该模型已经完成了基础闭环原型，但距离完全成熟的“穿透预警平台”仍有差距。

（一）主要成效

1. 已完成文件上传、模型勾选、任务启动、结果缓存和页面展示的基本闭环。
2. 已具备 11 个模型的注册、选择和批量运行入口。
3. 已具备统计结果、单模型结果和综合报告的标准化输出基础。
4. 为后续扩展跨模型关联链、风险图谱和更强的报告能力提供了工程框架。

（二）当前局限

1. 主数据映射基础仍需加强。
当前 DMP、SAP 及营销侧编码之间的统一主键体系尚需持续完善，这是跨模型联动的前提。

2. 授权清单和规则库治理机制仍需补充。
区域授权清单、红线规则库、客户名录等基础数据需建立动态维护和版本管理机制。

3. 关联链尚未真正产品化落地。
当前跨模型分析更适合作为风险提示和核查线索，尚未在代码中形成独立的关联链输出层和网页展示层，也不宜直接下因果判断。

4. 模型数量与落地深度之间需进一步平衡。
现阶段若同时推进全部 11 个模型的深度落地，容易出现“覆盖面广但深度不足”的问题。结合代码现状，更适宜优先做深 1.1、2.2、2.5 等业务价值更高、逻辑更完整的重点模型。

（三）优化方向

下一步建议优先聚焦业务价值高、规则边界清晰、数据条件相对成熟的重点模型，分阶段推进：

第一阶段，优先做深区域窜区检测、盈利偏差分析、施工真实性验证等模型。

第二阶段，补齐主数据映射规则库、授权清单治理机制和规则版本管理机制，并补充模型级前置数据校验。

第三阶段，再逐步扩展跨模型联动、关联提示结果表、关联链页面展示和更成熟的报告能力。

总体上，建议由“多模型全面铺开”调整为“重点模型先落地、基础治理同步补齐、关联能力后深化”的实施路径，以提升模型的可解释性、可信度和答辩说服力，并确保方案表述与当前代码能力保持一致。
